# WP Inside Live #01 → コーポレートサイト反映 設計書（マッピング表）

## 1. 目的と前提

### 1.1 目的

- **Inside Live というイベント名・体裁そのものはサイトに載せない**。
- `sandbox/inside-live_01` の要約は**素材（一次情報）**とし、各ページの読者（見込み顧客・採用候補）に必要な**事実・メッセージだけ**を既存ページへ分散反映する。
- 本書は「**どのソースから、どの URL／ページキーへ、何をどう書き換えて載せるか**」の設計・優先度・非公開範囲を定義する。

### 1.2 参照したサイト側メタ情報

- **正**: `public/data/pages.json` の各ページ `title` / `description` / `url` / `body`（AI 用要約含む）
- **ルーティング**: `public/index.php` の `$pageKey` と `docs/meta-registry.md`

## 3. マッピング表（ソース → 反映先）

### 記号

- **反映強度**: 高＝そのページの主メッセージに組み込みたい / 中＝1 ブロックまたは追記 / 低＝任意・1 文程度
- **公開区分**: ◎ そのまま一般向けに言い換え可能 / △ 抽象化・数値削除が必要 / ✕ 原則 Web 非掲載（内部・採用限定・クライアント承諾が必要）

## 4. 重複防止ルール（同一メッセージの扱い）

1. **AI・開発効率・請求モデル転換**は `business-web` を**主**、`business-operation` は運用契約の文脈で**1 回だけ**触れる。
2. **集客・見える化**は `business-marketing` と `services` で**役割分担**（受託の集客支援 vs メディア／マッチング）。
3. **ゲーム（Steam 含む）**は `business-game-dev-about-wp` に集約し、他ページは**受託ブラウザ**中心に留める。
4. **得意分野・組織**は `philosophy` を**主**、`message` は代表の言葉で**被りすぎない**よう調整。

## 5. `pages.json` 更新方針（AI・検索メタ）

文言を本文に反映したら、**同じ事実を `pages.json` の該当キーの `description` / `body` にも反映**する（`docs/meta-registry.md` の一元化ルール）。

優先して見直すキー候補:

- `business-web`（AI 開発・運用型請求・体制）
- `business-operation`（AI 活用運用の一文）
- `business-marketing`（施策の幅・誠実さの維持）
- `business-game-dev`, `business-game-dev-about-wp`（自社開発と受託の関係）
- `philosophy`（AI・スピード・得意分野の考え方）
- `services`（百人アンケートの鮮度）
